# logistics-ts v0.1 — Implementation Plan

_Based on `concept.md` + `research.md` (2026-07-11). This is the MVP plan: prove the gap, ship the agent surface, stay out of ERP territory._

---

## 1. Goals & non-goals

### Goals (v0.1 ships when all are true)

1. `npm i logistics-ts` — zero runtime dependencies, works in Node / browser / edge / Lambda unchanged.
2. Inventory math: safety stock (4 methods + auto), reorder point, EOQ, coverage metrics.
3. Classification: ABC, XYZ, FSN, ABC-XYZ matrix with policy hints, SBC demand classifier.
4. Forecasting: MA, SES, Holt (±damped), Holt-Winters (add/mult), Croston, SBA, TSB, `autoForecast`.
5. Every numeric output ships with a machine-readable **explanation** (method, inputs, reasoning).
6. Full AI-agent surface: TSDoc with formulas+citations, AGENTS.md, `skills/`, llms.txt, Context7 submission.
7. Synthetic data generator so agents/demos work with zero setup.

### Non-goals (explicitly out of v0.1)

- ARIMA (say so in docs; point at `arima` npm for now)
- DuckDB / Parquet adapters (`@logistics-ts/duckdb` is v0.2+; the `TableSource` seam ships now)
- Procurement/MOQ optimisation, multi-echelon, analytics/KPI package
- Any UI, dashboard, ERP connector (supplychainpy's death)
- Persistence — data in, results out, nothing stored

- **TypeScript strict**, ESM-only, `exports` maps with subpath exports per module (tree-shaking is a feature promise).
- Build: **tsdown** (or tsup) — dual `.d.ts` emit, no bundling of deps (there are none).
- Test: **vitest**. Lint/format: **biome** (single tool, fast).
- Versioning/publish: **changesets**, all packages fixed-versioned together for 0.x.
- CI: GitHub Actions — typecheck, test, publish on tag. Node 20/22/24 matrix + one browser (vitest browser mode or playwright smoke) to keep the "runs everywhere" promise honest.

Dependency direction (enforced with a lint rule or `dependency-cruiser`):

```
core ← inventory, classification, forecasting
inventory → forecasting (auto safety stock needs demand stats)  # one-way, allowed
logistics-ts (umbrella) → all
```

## 3. `@logistics-ts/core`

### 3.1 Data model

Canonical record types (all analytics speak these):

```ts
type DemandRecord = {
  itemId: string
  date: Date | string        // ISO accepted, normalised internally
  quantity: number
  locationId?: string
  unitPrice?: number         // needed for ABC by value
}

type StockRecord = {
  itemId: string
  quantity: number
  locationId?: string
  unitCost?: number
  timestamp?: Date | string
}

type LeadTimeRecord = {
  itemId: string
  leadTimeDays: number       // one row per receipt/PO, so σLT is computable
  date?: Date | string
}
```

### 3.2 Column store + loader

- Internal representation: dictionary-encoded string columns + `Float64Array` numeric columns, built once at load. This is an implementation detail — users never see it.
- `TableSource` interface: `{ numRows, columnNames, getColumn(name): ArrayLike }` — the seam future DuckDB/Parquet adapters implement.
- Loader with column mapping (the Prisma-mapping idea from the concept):

```ts
const demand = loadDemand(rows, {
  columns: { itemId: 'sku', date: 'txn_date', quantity: 'qty_shipped' },
})
// rows: object[] | TableSource | Record<string, ArrayLike> (columnar)
```

- Validation at load: missing columns, non-numeric quantities, unparseable dates → **collected issues** (`{ row, column, problem }[]`), fail-fast optional. Agents need actionable load errors.
- Time bucketing utility: `bucketize(demand, 'day' | 'week' | 'month')` → dense per-item series **with explicit zero-fill** (critical for intermittent-demand stats; document the zero-fill rule loudly).

### 3.3 Shared numerics (internal, exported for power users)

- `inverseNormalCdf(p)` — Acklam rational approximation (z-scores)
- `normalLossFunction(z)` — reserved for fill-rate in v0.2, cheap to include
- descriptive stats: mean, std (sample), CV, CV² of nonzero values, ADI
- **Nelder-Mead** minimiser (~80 lines) — used by Holt-Winters/SES parameter fitting

### 3.4 Explanation type (the differentiator — defined once, used everywhere)

```ts
type Explained<T> = {
  value: T
  method: string                       // e.g. "king-formula"
  inputs: Record<string, number | string>  // every number used
  reasoning: string[]                  // human/agent-readable bullets
  citations?: string[]                 // e.g. "King (2011), APICS Magazine"
  warnings?: string[]                  // e.g. "only 8 periods of history; low confidence"
}
```

### 3.5 Synthetic data

`generateExampleData({ items, periods, profile })` — profiles: `smooth`, `seasonal`, `intermittent`, `lumpy`, `trending`, `mixed` (default: a realistic mixed catalogue with lead times and prices). Seeded PRNG for reproducibility.

## 4. `@logistics-ts/classification`

| Export | Notes |
|---|---|
| `abc(items, { by: 'value' \| 'quantity', cutoffs?: [0.8, 0.95] })` | Returns class + cumulative share per item so callers can re-cut |
| `xyz(demand, { cutoffs?: [0.5, 1.0], bucket?: 'month' })` | CV-based; bucketing rule documented |
| `fsn(demand, { turnoverCutoffs? } )` | By turnover / last-movement recency |
| `abcXyzMatrix(abcResult, xyzResult)` | 9 cells, each with `policyHint` (AX: "automate, low SS" … CZ: "make-to-order") |
| `classifyDemandPattern(series)` | **SBC**: ADI + CV², cutoffs 1.32 / 0.49 → `smooth \| erratic \| intermittent \| lumpy`; option `classifier: 'sbc' \| 'kostenko-hyndman'` |

All results are `Explained<>`.

## 5. `@logistics-ts/forecasting`

### Methods (pure functions, uniform signature)

```ts
type ForecastResult = Explained<{
  forecast: { period: string; value: number }[]
  fitted?: number[]
  params: Record<string, number>       // fitted α, β, γ …
}>

movingAverage(series, { window })
ses(series, { alpha? })                // alpha omitted → fitted via Nelder-Mead on SSE
holt(series, { damped?: boolean })
holtWinters(series, { seasonLength, mode: 'additive' | 'multiplicative' })
croston(series, { alpha? })
sba(series, { alpha? })                // Croston × (1 − α/2)
tsb(series, { alphaDemand?, alphaProbability? })
seasonalDecompose(series, { seasonLength })   // classical MA-based
```

### Auto selection

```ts
autoForecast(series, { horizon, seasonLength? })
```

1. `classifyDemandPattern` (SBC).
2. Smooth/erratic → candidate set {SES, Holt, damped Holt, Holt-Winters if seasonality detected}; intermittent/lumpy → {Croston, SBA, TSB}. SBA is the intermittent default.
3. Rolling-origin backtest over candidates, select by **MASE**.
4. Explanation says exactly which quadrant, which candidates, which scores — this is the flagship "AI agents love explanations" feature.

### Metrics

`mae`, `rmse`, `mape` (warns on zeros), `smape`, `mase`, `bias`/tracking signal, plus `backtest(series, method, { origins })` (rolling-origin evaluator, reused by autoForecast).

## 6. `@logistics-ts/inventory`

```ts
safetyStock(input, { method: 'demand-variability' | 'leadtime-variability'
                           | 'king' | 'max-minus-average' | 'auto',
                     serviceLevel: number })
// 'auto' = recommendSafetyStockMethod(): uses SBC pattern + availability of
// σLT data to pick the formula — the concept's "method recommendation" feature

reorderPoint({ meanDemand, meanLeadTime, safetyStock })     // + order-up-to level
eoq({ annualDemand, orderCost, holdingCostPerUnit })        // + EPQ, quantity-discount variants
coverage(stock, demand)         // days of inventory, forecast-walk coverage
turnover(stock, demand)         // turns, DIO
issues(stock, demand, leadTimes, { serviceLevel })
// the concept's "issue analyser": per item flags —
// below-ROP, below-SS, stockout-risk-within-leadtime, overstocked (coverage > X),
// dead-stock (FSN=N with stock on hand). Returns Explained<Issue[]>.
```

All `Explained<>`, all TSDoc'd with formula + citation (King 2011; Silver-Pyke-Thomas; Harris 1913).

## 9. Testing & validation strategy

Trust in the numbers is the product. Three layers:

1. **Golden tests against reference implementations**: generate fixtures from Python `statsforecast` (Croston/SBA/TSB, ETS) and `stockpyl` (EOQ, safety stock) — checked-in JSON fixtures + a `fixtures/generate.py` script. Tolerance-based comparison.
2. **Textbook examples as tests**: worked examples from fpp3 (Holt-Winters) and Silver-Pyke-Thomas / King's paper (safety stock) reproduced exactly, cited in the test names.
3. **Property tests** (fast-check): SS monotonic in service level; EOQ scale invariance; ABC shares sum to 1; classifier total coverage; forecasts non-negative for non-negative series (where the method guarantees it).

Perf smoke test: 5M-row demand load + bucketize + full classify under a budgeted time (CI-tracked, generous bound).

## 11. Open decisions (resolve during M0/M1, defaults chosen)

1. **npm scope**: `@logistics-ts/*` requires owning the npm org — register it (and the bare `logistics-ts` name) **before writing code**. Fallback: `@logisticsjs/*`.
2. Sample vs population std dev for σD — default **sample** (n−1), document, allow override. Affects golden-test tolerances.
3. Date handling: no date library; internal epoch-day integers, ISO strings at the boundary. Timezone rule: dates are calendar dates, never instants.
4. `InventoryAnalyzer` class in umbrella only — core stays pure functions (keeps tree-shaking honest).
